# 실행계획을 이용한 쿼리 성능 비교

`EXPLAIN (ANALYZE, BUFFERS)`를 사용하여 동일한 결과를 반환하는 쿼리의 실행계획, 실제 실행 시간, 반복 횟수 및 버퍼 사용량을 비교하였다. `ANALYZE` 옵션은 쿼리를 실제로 실행하므로 본 비교에서는 데이터를 변경하지 않는 SELECT 문에만 적용하였다. 실행 시간은 캐시와 시스템 상태에 따라 달라질 수 있으므로 절대적인 값보다는 실행계획의 구조, `loops` 및 버퍼 접근량을 중심으로 분석하였다.

## 1. 미수강 학생 조회: NOT IN과 NOT EXISTS

두 쿼리는 모두 미수강 학생 100명을 반환하도록 작성하였다. `NOT IN` 방식에서는 서브쿼리 결과의 NULL로 인한 문제를 방지하기 위해 `student_id IS NOT NULL` 조건을 추가하였다.

| 비교 항목 | NOT IN | NOT EXISTS |
|---|---:|---:|
| 주요 계획 | Seq Scan + hashed SubPlan | Hash Anti Join |
| 실제 결과 행 | 100 | 100 |
| shared buffer hit | 21 | 21 |
| Execution Time | 2.229 ms | 0.582 ms |
| 예상 행 / 실제 행 | 500 / 100 | 98 / 100 |

`NOT IN`은 `enroll`을 읽어 해시 서브플랜을 만든 뒤 `student`의 각 행을 필터링하였다. `NOT EXISTS`는 PostgreSQL이 이를 `Hash Anti Join`으로 변환하여 두 테이블을 직접 비교하였다. 이번 실행에서 두 쿼리의 shared buffer hit는 21회로 같았지만, `NOT EXISTS`의 실행 시간이 약 3.8배 짧았으며 예상 행 수도 실제 100명에 더 가까웠다.

작은 데이터에서는 실행 시간의 절대 차이가 크지 않지만, `NOT EXISTS`는 서브쿼리 결과에 NULL이 포함되어도 의미가 변하지 않는다는 장점이 있다. 따라서 미존재 여부를 판단하는 안티 조인에서는 성능뿐 아니라 정확성 측면에서도 `NOT EXISTS`를 우선 사용하는 것이 적절하다.

## 2. 학과 평균 초과 학생 조회: 상관 서브쿼리와 사전 집계 JOIN

두 쿼리는 모두 자신의 학과 평균 GPA보다 높은 학생 495명을 반환한다. 첫 번째 쿼리는 학생마다 학과 평균을 계산하는 상관 서브쿼리이며, 두 번째 쿼리는 학과 평균을 먼저 계산한 뒤 학생 테이블과 JOIN한다.

| 비교 항목 | 상관 서브쿼리 | 사전 집계 CTE + JOIN |
|---|---:|---:|
| 주요 계획 | Seq Scan + 반복 SubPlan | HashAggregate + Hash Join |
| 실제 결과 행 | 495 | 495 |
| 평균 계산 반복 횟수 | 1,000회 | 1회 |
| shared buffer hit | 8,008 | 16 |
| Execution Time | 77.930 ms | 2.608 ms |

상관 서브쿼리는 외부 `student`의 1,000개 행마다 내부 평균 계산을 다시 실행하였다. 실행계획에서도 내부 `Aggregate`와 `Seq Scan`의 `loops=1000`을 확인할 수 있으며, 그 결과 shared buffer hit가 8,008회 발생하였다.

사전 집계 방식은 `HashAggregate`로 6개 학과의 평균을 한 번 계산한 뒤 `Hash Join`으로 학생과 연결하였다. 평균 계산의 반복을 제거하여 shared buffer hit가 16회로 감소했고, 이번 실행에서는 약 2.608ms에 완료되었다. 상관 서브쿼리의 77.930ms와 비교하면 실행 시간이 약 30배 짧았다.

상관 서브쿼리는 문법과 동작 방식을 학습하기에는 적합하지만, 동일한 집계를 행마다 반복할 수 있다. 따라서 실제 대용량 조회에서는 반복 계산 여부를 실행계획의 `loops`로 확인하고, 사전 집계 CTE 또는 JOIN으로 변경하는 방안을 검토해야 한다.

## 종합 분석

실행계획 비교를 통해 SQL 문장이 동일한 결과를 반환하더라도 내부 처리 방식과 비용은 크게 달라질 수 있음을 확인하였다. 미존재 조회에는 NULL에 안전하고 Anti Join으로 최적화될 수 있는 `NOT EXISTS`가 적합했다. 반복 집계를 포함한 상관 서브쿼리는 가독성과 학습 목적에는 유용하지만, 데이터가 커질수록 반복 실행 비용이 증가하므로 사전 집계와 JOIN 방식이 더 효율적이었다.

성능을 판단할 때는 `Execution Time` 하나만 비교하지 않고 다음 항목을 함께 확인해야 한다.

- 계획 노드: Seq Scan, Hash Join, Hash Anti Join, HashAggregate 등
- 실제 행과 예상 행: `actual rows`와 `rows`
- 반복 횟수: `loops`
- 버퍼 사용량: `Buffers: shared hit/read`
- 정렬 방식과 메모리 사용량
- Planning Time과 Execution Time

실행 시간은 캐시 상태와 장비 환경에 따라 변할 수 있으므로 여러 번 실행하여 대표값을 확인하고, 실행계획의 구조적인 차이를 중심으로 해석하는 것이 중요하다.
