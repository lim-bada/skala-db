# 종합실습2 보고서 설명문

이 문서는 `종합실습2초안.pdf`에 배치된 1~25번 SQL 실행 화면 아래에 삽입할 설명문이다. 코드와 실행 결과는 캡처로 제시하고, 본문에서는 쿼리의 목적, 적용한 SQL 문법 및 결과의 의미를 설명한다.

## 1. 실습 환경 및 진행 기준

PostgreSQL의 `skala_db` 데이터베이스에 `skala_user`로 접속하고, `search_path`를 `lab, public`으로 설정한 상태에서 실습을 진행하였다. 제공된 SQL 파일로 생성된 `student`, `enroll`, `customers`, `orders`, `emp` 테이블을 사용했으며, 사전 점검을 통해 각 테이블의 구조와 데이터 건수를 확인하였다.

실습 문항과 작성해야 할 SQL의 종류는 수정된 `종합실습2.pdf`를 기준으로 하였다. 다만 실제 조회 대상 데이터는 제공된 SQL 파일을 실행해 생성된 원본 데이터를 그대로 사용하였다. 따라서 PDF의 데이터 생성 규칙과 제공 SQL의 실제 생성 결과가 다른 부분에서는 데이터를 임의로 변경하지 않고, 실제 테이블에 저장된 값을 기준으로 쿼리를 실행하고 결과를 해석하였다.

## 2. 제공 PDF와 SQL 파일의 데이터 규칙 차이

### 2.1 학생 수강 데이터

수정된 PDF에는 학생 번호를 3으로 나눈 나머지에 따라 학생별 수강 건수가 0건, 1건, 2건이 되도록 구성한다고 설명되어 있다. 이 규칙을 학생 1,000명에게 적용하면 미수강 학생 333명, 1과목 수강 학생 334명, 2과목 수강 학생 333명이 생성된다. 여기에 학생 테이블에 존재하지 않는 `student_id` 1001과 1010의 고아 수강 2건을 더하면 예상 `enroll` 데이터는 총 1,002건이다.

그러나 제공된 SQL 파일을 실제로 실행한 결과는 다음과 같았다.

| 구분 | PDF 설명 기준 | 제공 SQL 실행 결과 |
|---|---:|---:|
| 전체 학생 | 1,000명 | 1,000명 |
| 미수강 학생 | 333명 | 100명 |
| 1과목 수강 학생 | 334명 | 0명 |
| 2과목 수강 학생 | 333명 | 400명 |
| 3과목 수강 학생 | 0명 | 500명 |
| 고아 수강 | 2건 | 2건 |
| 전체 수강 데이터 | 1,002건 | 2,302건 |

제공 SQL에서는 학생 번호가 10의 배수인 학생에게 수강 데이터가 없고, 그 외 학생은 조건에 따라 2건 또는 3건의 수강 데이터가 생성된다. 따라서 실습 결과에서 미수강 학생이 `10, 20, 30, ...` 순서로 나타나고, 1과목만 수강한 학생이 존재하지 않는 것은 쿼리 오류가 아니라 제공 SQL의 생성 규칙에 따른 결과이다.

### 2.2 조직 계층 데이터

PDF에서는 `CEO → 매니저 10명 → 개발자 300명`의 3단계 조직 구조를 설명한다. 하지만 제공 SQL에서는 매니저가 `emp_id` 2~11로 생성된 반면, 개발자의 `manager_id`는 1~10으로 배정되었다.

그 결과 CEO(`emp_id=1`)에게 개발자 30명이 직접 연결되고, `Mgr_2`부터 `Mgr_10`까지는 각각 개발자 30명이 연결되며, `Mgr_11`의 직속 부하 직원은 0명으로 조회된다. 이 부분 역시 데이터를 별도로 보정하지 않고 제공 SQL로 생성된 원본 상태를 유지하였다. 따라서 재귀 CTE와 직속 부하 집계 쿼리의 문법은 PDF의 요구사항에 맞게 작성하되, 결과 해석은 실제 데이터 구조를 기준으로 하였다.

## 3. 실습별 결과 설명

### 1. 학생과 수강을 INNER JOIN하여 수강 존재 학생의 과목과 성적 조회

`student`와 `enroll`을 `student_id`로 INNER JOIN하였다. INNER JOIN은 양쪽 테이블에 연결되는 데이터가 모두 존재하는 행만 반환하므로, 수강 기록이 있는 학생의 학번, 이름, 과목 및 성적만 조회된다. 학생 번호와 과목 순으로 정렬하여 한 학생의 여러 수강 내역을 연속해서 확인할 수 있도록 하였다.

### 2. 모든 학생을 기준으로 수강 정보 연결

`student`를 기준 테이블로 두고 `enroll`을 LEFT JOIN하였다. 수강 기록이 없는 학생도 결과에서 제외되지 않으며, 해당 학생의 과목과 성적은 `NULL`로 표시된다. 실행 결과의 `Student_1000`은 학생 정보는 존재하지만 수강 정보가 없어 과목과 성적이 `NULL`로 출력된 사례이다.

### 3. 수강을 기준으로 학생 정보 연결

RIGHT JOIN을 사용하여 `enroll`의 모든 수강 기록을 유지하면서 학생 정보를 연결하였다. 학생 테이블에 대응하는 학번이 없으면 학생 이름이 `NULL`로 표시된다. 실행 결과에서 `student_id` 1001과 1010은 `enroll`에만 존재하는 고아 수강 데이터이므로 학생 이름이 `NULL`로 조회되었다.

### 4. 학생과 수강 양쪽의 모든 데이터 조회

FULL OUTER JOIN을 사용하여 학생과 수강 양쪽의 모든 행을 포함하였다. 두 테이블에 모두 존재하는 정상 수강 데이터뿐 아니라, 수강 기록이 없는 학생과 학생 정보가 없는 고아 수강도 함께 조회된다. 학번은 `COALESCE(student.student_id, enroll.student_id)`로 표시하여 어느 한쪽에만 학번이 있어도 결과에서 식별할 수 있도록 하였다.

### 5. 한 번도 수강하지 않은 학생 조회

상관 서브쿼리와 `NOT EXISTS`를 사용하여 현재 학생과 연결되는 수강 행이 존재하지 않는 경우만 조회하였다. `NOT EXISTS`는 NULL의 영향을 받을 수 있는 `NOT IN`보다 안전한 안티 조인 패턴이다. 실제 데이터에서는 학생 번호가 10의 배수인 학생이 미수강 학생으로 조회되었다.

### 6. 한 과목 이상 수강한 학생 조회

`EXISTS`를 사용하여 각 학생에게 하나 이상의 수강 기록이 존재하는지 확인하였다. 존재 여부만 판단하므로 여러 과목을 수강한 학생도 결과에는 한 번만 출력되며, 별도의 `DISTINCT`가 필요하지 않다. 이는 수강 테이블의 열을 출력하지 않고 수강 여부만 확인할 때 적합한 세미 조인 패턴이다.

### 7. 고객별 주문 건수와 총 주문 금액 조회

`customers`와 `orders`를 고객 번호로 연결한 뒤 고객별로 그룹화하였다. `COUNT(order_id)`로 주문 건수를 계산하고 `SUM(amount)`로 총 주문 금액을 계산하였다. 제공 데이터에서는 모든 고객이 주문을 6건씩 가지고 있으므로 INNER JOIN을 사용해도 전체 고객의 집계 결과가 정상적으로 출력된다.

### 8. 주문 총액 상위 10명 조회

고객별 주문 금액을 합산한 후 총액을 내림차순으로 정렬하고 `LIMIT 10`을 적용하였다. 총액이 같은 경우에도 결과 순서가 일정하도록 고객 번호를 보조 정렬 기준으로 사용하였다. 실행 결과에서는 `Customer_154`가 가장 높은 총 주문 금액을 가진 고객으로 조회되었다.

### 9. 모든 직원과 매니저 이름 조회

하나의 `emp` 테이블을 직원 역할과 매니저 역할로 각각 별칭을 지정하여 SELF JOIN하였다. 모든 직원을 유지하기 위해 LEFT JOIN을 사용했으므로 상위 관리자가 없는 CEO도 결과에 포함되며, CEO의 매니저 이름은 `NULL`로 표시된다.

### 10. 모든 학생 기준 과목별 수강 인원 집계

모든 학생을 유지하기 위해 `student`를 기준으로 `enroll`을 LEFT JOIN하였다. 과목별로 학생 수를 `COUNT(DISTINCT student_id)`로 계산하여 동일 학생의 중복 집계를 방지하였다. 수강 기록이 없는 학생은 `COALESCE`로 과목명을 `미수강`으로 표시했으며, 실제 제공 데이터에서는 미수강 학생이 총 100명으로 집계되었다.

### 11. DB 과목을 수강하지 않은 학생 조회

`NOT EXISTS` 서브쿼리 내부에서 현재 학생의 학번과 `course = 'DB'` 조건을 함께 검사하였다. 따라서 다른 과목만 수강한 학생과 한 과목도 수강하지 않은 학생이 모두 포함되며, DB 과목 수강 기록이 하나라도 있는 학생은 제외된다.

### 12. 과목별 운영 책임 매니저 구성 및 리포트 작성

#### 12.1 course_owner 테이블 생성

과목과 담당 매니저의 관계를 저장하기 위해 `course_owner` 테이블을 생성하였다. 과목명은 중복될 수 없도록 기본 키로 지정하고, `manager_id`는 반드시 값이 존재하도록 `NOT NULL` 제약조건을 적용하였다.

#### 12.2 과목별 매니저 임의 배정

수강 테이블의 고유 과목과 이름이 `Mgr_`로 시작하는 매니저에게 각각 `ROW_NUMBER()`를 부여하였다. 과목 순번에 나머지 연산을 적용해 10명의 매니저가 23개 과목을 순환해서 담당하도록 구성하고, 그 결과를 `course_owner`에 입력하였다. 입력 결과 23개의 과목-매니저 매핑이 생성되었다.

#### 12.3 과목별 수강 인원과 책임 매니저 조회

`course_owner`를 기준으로 `enroll`과 `emp`를 연결하였다. 과목별 고유 학생 수를 집계하고 담당 매니저 이름을 함께 출력하여 과목 운영 현황 리포트를 구성하였다. 과목을 기준으로 정렬하여 각 과목의 수강 인원과 책임자를 쉽게 비교할 수 있도록 하였다.

### 13. 학생별 과목 추천 후보 생성

학생과 고유 과목 목록을 CROSS JOIN하여 가능한 모든 학생-과목 조합을 생성하였다. 현재 데이터에는 학생 1,000명과 고유 과목 23개가 있으므로 전체 조합은 23,000건이다. CROSS JOIN은 결과가 급격히 증가할 수 있으므로 문제에서 요구한 대로 정렬 후 `LIMIT 100`을 적용하여 샘플만 확인하였다.

### 14. SELECT 절의 스칼라 서브쿼리

각 학생 행에 대해 SELECT 절의 스칼라 서브쿼리로 동일 학생의 `major`를 조회하였다. 기본 키인 `student_id`로 조건을 지정했기 때문에 서브쿼리는 한 행의 한 값만 반환하며 스칼라 서브쿼리의 조건을 만족한다. 이 문항은 스칼라 서브쿼리의 동작을 확인하기 위한 것으로, 실제 업무에서는 현재처럼 같은 테이블의 값을 조회할 경우 외부 쿼리의 `s.major`를 직접 사용하는 편이 효율적이다.

### 15. 전체 평균 GPA보다 높은 학생 조회

비상관 서브쿼리에서 전체 학생의 평균 GPA를 한 번 계산하고, 외부 쿼리에서 각 학생의 GPA와 비교하였다. 평균보다 높은 학생만 남긴 후 GPA 내림차순과 학생 번호 오름차순으로 정렬하였다.

### 16. 자신의 학과 평균 GPA보다 높은 학생 조회

상관 서브쿼리에서 외부 학생의 `major`를 참조하여 해당 학과의 평균 GPA를 계산하였다. 학생마다 자신이 소속된 학과의 평균과 비교하므로 전체 평균을 사용하는 15번과 비교 기준이 다르다. 상관 서브쿼리의 행별 평가 방식을 확인하기 위한 쿼리이며, 대용량 환경에서는 학과 평균을 먼저 집계한 CTE 또는 JOIN 방식도 고려할 수 있다.

### 17. 수강 기록이 있는 학생 조회

`EXISTS`를 사용하여 현재 학생과 연결된 수강 기록의 존재 여부를 판단하였다. 서브쿼리에서는 실제 값을 반환할 필요가 없으므로 관례적으로 `SELECT 1`을 사용하였다. 수강 과목 수와 관계없이 학생별로 한 행만 반환된다.

### 18. 한 번도 수강하지 않은 학생 조회

`NOT EXISTS`를 사용하여 현재 학생과 일치하는 수강 기록이 없는 학생만 조회하였다. 17번의 조건을 반대로 적용한 형태로, 세미 조인과 안티 조인의 차이를 확인할 수 있다. 실행 결과는 실제 데이터 생성 규칙에 따라 학생 번호 10, 20, 30 등의 순서로 나타났다.

### 19. HR 학과 학생 일부와의 GPA 비교

`> ANY`를 사용하여 현재 학생의 GPA가 HR 학과 학생들의 GPA 중 하나 이상보다 높은지 비교하였다. HR 학과의 비교 대상이 존재하는 현재 데이터에서는 실질적으로 HR 학과 GPA의 최솟값보다 높은 학생을 찾는 조건으로 동작한다. `ANY`는 모든 값보다 커야 하는 `ALL`과 의미가 다르다.

### 20. CS 학과 또는 DB 과목 수강 학생 조회

첫 번째 SELECT에서 CS 학과 학생을 조회하고, 두 번째 SELECT에서 DB 과목 수강 학생을 조회한 뒤 `UNION`으로 결합하였다. 한 학생이 두 조건을 모두 만족할 수 있으므로 중복을 제거하는 `UNION`을 사용하였다. 중복을 유지하는 `UNION ALL`과의 차이를 확인할 수 있는 문항이다.

### 21. 학과별·GPA 구간별 인원과 소계·총계 조회

CTE에서 CASE 식을 사용하여 GPA를 `3.0 미만`, `3.0~3.5`, `3.5 초과`의 세 구간으로 분류하였다. `ROLLUP(major, gpa_tier)`을 적용하여 학과별 구간 인원, 학과별 소계 및 전체 총계를 하나의 쿼리로 계산하였다. `GROUPING()`으로 집계 과정에서 생성된 NULL을 구분하여 `소계`, `전체`, `총계` 라벨을 표시하였다. 결과의 전체 총계는 학생 테이블의 전체 건수와 동일한 1,000명으로 확인되었다.

### 22. 재귀 CTE를 이용한 조직 계층 조회

CEO처럼 `manager_id IS NULL`인 최상위 행을 기준 행으로 정하고, 재귀 행에서 현재 직원의 `emp_id`와 하위 직원의 `manager_id`를 연결하였다. 재귀가 한 단계 진행될 때마다 `depth`를 1씩 증가시키고 이름을 이어 붙여 조직 경로인 `path`를 생성하였다. 비정상적인 순환 관계가 있을 때 무한히 탐색하는 것을 방지하기 위해 최대 깊이를 10으로 제한하였다.

별도의 SELF JOIN 집계에서는 매니저를 기준으로 직속 부하 직원을 LEFT JOIN하고 `COUNT(employee.emp_id)`로 인원수를 계산하였다. LEFT JOIN을 사용했기 때문에 직속 부하가 없는 `Mgr_11`도 0명으로 결과에 포함되었다. `Mgr_11`이 0명으로 나타나는 것은 앞에서 설명한 제공 SQL의 매니저 번호 배정 차이에 따른 결과이다.

### 23. 학과별 GPA 상위 3명 및 순위 함수 비교

학과별로 파티션을 나누고 GPA 내림차순으로 순위를 계산하였다. `ROW_NUMBER()`에는 학생 번호를 보조 정렬 기준으로 추가하여 GPA가 같은 학생도 1, 2, 3처럼 고유한 순번을 갖도록 하였다. 반면 `RANK()`와 `DENSE_RANK()`는 GPA만 기준으로 계산하여 동점 학생에게 같은 순위를 부여하고, 다음 순위의 건너뜀 여부 차이를 비교할 수 있도록 하였다.

`COUNT(*) OVER(PARTITION BY major)`로 학과별 전체 학생 수도 각 행에 함께 표시하였다. 계산된 순번은 같은 SELECT 문의 WHERE 절에서 바로 사용할 수 없으므로, 먼저 인라인 뷰 또는 CTE에서 순위를 계산하고 외부 쿼리에서 `row_num <= 3`을 적용하였다. 두 방식은 구조만 다르고 동일하게 학과별 상위 3명씩 총 18명을 반환하였다.

### 24. LAG를 이용한 과목 간 성적 변화 분석

첫 번째 CTE에서 CASE 식으로 문자 성적을 A=4, B=3, C=2, D=1의 숫자 점수로 변환하였다. 두 번째 CTE에서는 학생별로 파티션을 나누고 과목 순서에 따라 `LAG(score)`로 이전 과목의 점수를 가져왔다. 현재 점수와 이전 점수의 차이를 `diff`로 계산하고, 결과에 따라 `상승`, `유지`, `하락`으로 분류하였다.

학생의 첫 번째 과목은 비교 대상이 없으므로 이전 점수와 차이가 `NULL`이며 `비교 없음`으로 표시하였다. 또한 학생별 `MAX(score) - MIN(score)`를 윈도우 함수로 계산하여 개인별 최고점과 최저점의 차이를 `score_range`로 출력하였다.

### 25. 주문 누적합, 이동평균 및 50% 초과 지점 조회

주문을 `order_id` 순으로 정렬하고 `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` 프레임을 사용하여 첫 주문부터 현재 주문까지의 누적 주문 금액을 계산하였다. `ROWS BETWEEN 2 PRECEDING AND CURRENT ROW`로 현재 행을 포함한 최근 3개 주문의 이동평균을 계산하였으며, 고객별 파티션을 적용한 누적합도 함께 출력하였다.

추가 CTE에서는 전체 주문 금액과 주문 순서별 누적 금액을 계산한 뒤, 누적 금액이 전체 금액의 50%를 초과하는 행만 남기고 가장 앞선 주문 한 건을 조회하였다. 전체 금액의 절반은 3,682,992.50이며, `order_id` 1502까지의 누적 금액은 3,682,041.53으로 절반 이하이다. 다음 주문인 `order_id` 1503에서 누적 금액이 3,686,585.56이 되어 처음으로 50%를 초과하므로 최종 결과는 1503으로 확인되었다.

## 4. 쿼리 작성 원칙 및 최적화 고려사항

- 필요한 컬럼만 명시하고 `SELECT *`을 사용하지 않았다.
- 테이블과 컬럼에 의미가 분명한 별칭을 적용하고, 모든 일반 JOIN에 연결 조건을 명시하였다.
- 외부 조인의 기준 행을 유지하고, 조인 이후의 조건 때문에 INNER JOIN처럼 바뀌지 않도록 조건 위치를 구분하였다.
- 존재 여부 판단에는 `EXISTS`, 미존재 여부에는 NULL에 안전한 `NOT EXISTS`를 사용하였다.
- 집계에서는 `COUNT(*)`, `COUNT(컬럼)`, `COUNT(DISTINCT 컬럼)`의 NULL 및 중복 처리 차이를 고려하였다.
- 소계와 총계에는 `ROLLUP`과 `GROUPING`을 사용하고, 집계되지 않은 일반 컬럼을 잘못 SELECT하지 않도록 GROUP BY 기준을 일치시켰다.
- 재귀 CTE에는 종료 조건을 두고, 윈도우 누적 계산에는 `ROWS` 프레임을 명시하였다.
- 중복 제거가 필요한 집합 결합에는 `UNION`을 사용하고, 의도적인 CROSS JOIN에는 결과 확인용 `LIMIT`을 적용하였다.
- 출력 결과가 반복 실행 시 임의의 순서로 달라지지 않도록 주요 조회에 `ORDER BY`와 보조 정렬 기준을 적용하였다.
- 제공 스크립트의 `enroll(student_id)`, `orders(customer_id)`, `emp(manager_id)` 인덱스는 주요 JOIN과 상관 서브쿼리의 연결 조건을 지원한다. 데이터가 더 커질 경우 `enroll(course)`, `enroll(student_id, course)`, `student(major, gpa, student_id)` 등의 복합 인덱스를 후보로 검토할 수 있으나, 실제 추가 여부는 `EXPLAIN ANALYZE`로 실행 계획과 비용을 비교한 후 결정하는 것이 적절하다.

## 5. 종합 결과

INNER·OUTER·SELF·CROSS JOIN, 집계, 서브쿼리, 집합 연산, CTE, 재귀 CTE 및 Window Function을 단계적으로 적용하였다. 모든 쿼리는 수정된 종합실습2 문항의 요구사항을 기준으로 작성하였으며, 실행 결과는 제공 SQL 파일로 생성된 실제 데이터에 따라 검증하였다.

PDF의 설명과 제공 SQL의 데이터 생성 규칙 사이에 수강 건수 및 조직 계층 배정 차이가 있었지만, 데이터 원본을 임의로 변경하지 않고 차이를 명시한 뒤 실제 저장된 데이터를 기준으로 결과를 해석하였다. 이를 통해 문항의 SQL 문법을 충족하는 것뿐 아니라, 데이터의 전제와 실제 상태를 검증하고 그 차이를 설명하는 과정도 함께 수행하였다.
